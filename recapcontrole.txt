Cause n°2 — même un <style> bien sauvegardé ne change presque rien
Dans backend/src/lib/email/layout.ts, tout le design visible est en styles inline : le gradient bleu du header, le fond de carte, les border-radius, paddings, couleurs, footer… tout est écrit en style="…" sur les balises. Le bloc <style> (DEFAULT_STYLE) ne contient que : le reset CSS, les :hover et les @media (responsive + dark mode).
Deux conséquences :

Spécificité CSS : un style inline bat toujours un sélecteur de classe. Si tu ajoutes .card-header{background:red} dans le <style>, ça ne surchargera jamais le style="background:linear-gradient(...)" inline du même élément (sauf !important).
Clients mail : Gmail (surtout mobile) et Outlook ignorent ou strippent largement le <head><style>. C'est précisément pour ça que le layout a été écrit en inline… ce qui rend l'édition du <style> inopérante en pratique.

Autrement dit : la seule chose que tu peux réellement modifier via cet éditeur aujourd'hui, c'est le hover / le dark mode / le responsive — pas l'apparence de base.
Ce qu'il faut décider pour corriger
Le vrai problème est architectural : on ne peut pas personnaliser un design inline en surchargeant un <style>. Trois directions possibles, par ambition croissante :

Rapide / honnête — assumer que seul le <style> est éditable : dans layout.ts, remplacer les styles inline structurants par des classes (.card, .card-header, .cta…) définies dans DEFAULT_STYLE, avec !important sur les surcharges. L'éditeur devient alors réellement fonctionnel. Compromis : légère perte de compat Outlook sur certains points.

C'est robuste sur tous les clients mail et bien plus simple à utiliser côté admin.
Full custom — stocker le HTML complet du layout en base (pas juste le <style>) avec des placeholders {{content}}, {{preheader}}… et laisser baseLayout() l'hydrater. Puissant mais tu déportes la responsabilité de la compat email sur l'éditeur admin (risqué).

Nuance : si tu passes ton JWT uniquement en header Authorization: Bearer (jamais en cookie auto-envoyé), le CSRF classique ne s'applique plus — c'est une alternative valable à csrf-csrf.

file-type — tu utilises multer pour les uploads d'images (rehostImage). Ne fais jamais confiance au mime-type envoyé par le client : file-type lit les magic bytes du fichier pour vérifier que c'est vraiment une image. Bloque l'upload d'un .php/.svg piégé déguisé en .png.

Utiles, peu coûteux
hpp — protège contre la pollution de paramètres HTTP (?role=user&role=admin). Un middleware, une ligne, aucun inconvénient.

envalid (ou juste ton zod existant) — valide tes variables d'env au démarrage. Aujourd'hui si JWT_SECRET est absent en prod, ça peut démarrer silencieusement avec un secret vide → catastrophe. Fais échouer le boot bruyamment.

Ton express-rate-limit actuel stocke les compteurs en mémoire : sur plusieurs instances Railway, la limite devient contournable (chaque instance compte séparément).
